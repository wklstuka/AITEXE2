2 Vibe Coding 概念深度解析

Vibe Coding = 清晰的意图描述 + AI 代码生成 + 人类审查迭代
维度 传统编程 Vibe Coding
核心技能 语法记忆 + 算法设计 需求表达 + 代码审查
开发速度 慢（逐行编写） 快（批量生成）
最大瓶颈 编码能力 需求表达能力 + 审查能力

三大误区 ❌

  • 完全不需要懂代码（你需要能读懂）
  • AI 不会出 Bug（它的 Bug 更隐蔽）
  • 只能做 Demo（可做生产级项目）

五大核心技巧 ✅

  • Spec 式 Prompt (产品文档化)
  • 分步迭代 (Step by Step)
  • 上下文管理 (Context Is King)
  • 约束边界 (不要做什么)
  • 持续 Review (不盲目 Accept)

Boris Cherny（Claude Code 创始人、Anthropic Head of Claude Code）于 2026 年 1 月在 X 分享其团队实际使用 Claude Code 的 10 条原则，获得 850 万+ 浏览。这些原则适用于所有 AI 辅助编程场景。

  1. 并行工作 — 启动 3–5 个 git worktree，每个运行独立的 Claude 会话，成倍提升吞吐量

  3. 投资 CLAUDE.md — 每次修正后让 AI 更新记忆文件：「Update your CLAUDE.md so you don't make that mistake again」
  4. 创建 Skills 并提交 Git — 团队共享工作流，常用操作做成 slash command，如 /techdebt

  7. 终端与环境 — 自定义 statusline、Post-tool hooks 自动格式化、/permissions 预放行常用命令

  10. 用 AI 学习 — 生成可视化 HTML 讲解陌生代码；启用 Learning 输出风格，让 AI 解释「为什么」

4 Agent 与 Sub-Agent：AI IDE 的"大脑" ⭐ 重点

4.1 什么是 Agent（智能体）

一句话定义：Agent 不只是「回答问题」，它能自主规划、执行步骤、使用工具、观察结果、迭代修正。

维度 普通 Chat Agent 模式
做什么回答一个问题完成一个任务
能否操作文件❌ 只输出文本✅ 创建/编辑/删除文件
能否运行命令❌✅ 执行终端命令
能否搜索代码❌✅ 搜索整个项目
多步骤推理有限✅ 制定计划 → 逐步执行
错误处理需你手动修✅ 自动检测并修复

4.2 ReAct 工作循环（Think → Act → Observe）

ReAct = Reasoning + Acting，让 AI 在推理中交替输出「思考」和「行动」，利用环境反馈不断迭代。

1
Thought（思考）：规划下一步、整理已知信息、解释为何调用工具
2
Action（行动）：发出结构化命令，触发文件操作、终端、搜索等
3
Observation（观察）：记录环境反馈，为下一步推理提供依据
Todo App 实战示例
Think: "需要 HTML/CSS/JS 三个文件"
Act:   创建 index.html，写入结构
Observe: 文件创建成功 ✓

Think: "接下来写样式"
Act:   创建 style.css
Observe: 完成 ✓

Think: "写逻辑并测试"
Act:   创建 app.js + 运行 live-server
Observe: 页面正常渲染 ✅

4.3 Agent 的工具库（Tools）

Agent 不是「空想家」，它有一套真实的「武器库」：

📁 文件操作
创建、读取、编辑、删除
⌨️ 终端命令
npm install, git commit
🔍 代码搜索
查找引用、符号、跨文件
🌐 浏览器
打开网页、截图、交互

4.4 什么是 Sub-Agent（子智能体）

复杂任务时，主 Agent 派生子任务给 Sub-Agent 独立执行。

为什么需要 Sub-Agent？
  • 🧠 上下文窗口有限，子任务拆分后各自独立
  • 🎯 专注度更高，每 Agent 只做一件事
  • ⚡ 可并行执行，加速开发
  • 🛡️ 风险隔离，一个失败不影响其他
核心隐喻
主 Agent = 项目经理（统筹、分配）
Sub-Agent = 专项工程师（专注执行）

用户: "做一个博客，有列表、详情、评论"

主 Agent:
├─→ Sub-Agent 1: 列表页 → ArticleList.jsx
├─→ Sub-Agent 2: 详情页 → ArticleDetail.jsx
├─→ Sub-Agent 3: 评论组件 → Comments.jsx
└── 汇总 → 整合路由 → 测试 ✅

4.6 最佳实践（Do's & Don'ts）

✅ DO
  • • 给 Agent 清晰、完整的任务描述（像写 Spec）
  • • 让 Agent 先制定计划，你确认后再执行
  • • 复杂任务拆成多个独立 Agent 会话
  • • 每次 Agent 完成后审查代码 Diff
  • • 用 @file 明确指定相关文件，减少噪音
❌ DON'T
  • • 一次给 Agent 太多不相关任务
  • • 让 Agent 修改你不了解的核心代码
  • • 不看代码就盲目 Accept All
  • • 在一个超长会话里做所有事（上下文溢出）
  • • 忽略 Agent 的中间计划，直接催结果

6 MCP (Model Context Protocol)

6.1 什么是 MCP？

MCP 是模型上下文协议，由 Anthropic 等推动的开放协议，让大模型与外部数据源、工具建立标准化通信。可类比为：AI 连接外部世界的「USB 接口」。

宿主(IDE) ←→ MCP Client ←→ MCP Server ←→ 外部服务(GitHub/DB/浏览器...)
Prompts
结构化提示模板，参数化复用
Resources
结构化数据（文件、API），URI 标识
Tools
可被 LLM 发现和调用的外部工具

6.2 如何配置 MCP？

在 Cursor / Claude Desktop 的配置文件中添加 MCP Server。以 Cursor 为例，编辑 ~/.cursor/mcp.json：

{ "mcpServers": { "playwright": { "command": "npx", "args": ["@playwright/mcp@latest"] }, "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"] } } }

配置后重启 IDE，Agent 即可自动发现并调用这些工具。

6.3 使用案例

案例 1：Playwright MCP — 浏览器自动化

Microsoft 官方提供，让 AI 控制真实浏览器。基于无障碍快照（非截图），轻量且可靠。

应用场景：生成 Playwright 测试代码、自动化填表、网页内容爬取、探索网站结构
Prompt: "请用 Playwright 打开 https://example.com，截图首页，并提取页面中所有链接的文本和 URL，输出为 JSON。"
案例 2：GitHub MCP — 仓库与协作自动化

GitHub 官方 MCP Server，让 AI 管理仓库、Issue、PR、读取远程代码。

应用场景：创建 PR、搜索 Issue、分析仓库结构、生成 Release Notes
Prompt: "请用 GitHub MCP 读取仓库 facebook/react 的 README 和 package.json，总结该项目的技术栈和主要依赖，并给出一个适合贡献者的入门建议。"

常用 MCP Server 推荐

@playwright/mcp @modelcontextprotocol/server-github filesystem fetch postgres

7 Skill 系统

7.1 什么是 Skill？

Skill = 预编写的指令集 + 资源包，教 AI「如何做某类特定任务」。类比：MCP 是工具箱，Skill 是操作手册（SOP）。

典型 Skill 文件结构
.skills/
└── ui-design/
    ├── SKILL.md          # 核心指令（必须）
    ├── scripts/          # 辅助脚本
    ├── examples/         # 参考设计截图
    └── resources/        # 色板、字体、模板

7.2 应用方法

  • 1. 在项目根目录创建 .skills/<skill-name>/SKILL.md
  • 2. 在 SKILL.md 中写明：身份、能力、约束、输出格式、示例
  • 3. 在 Cursor/Trae 中用 @skill 或 @.skills/skill-name 引用
  • 4. Prompt 中直接描述任务，AI 会按 Skill 规范执行

7.3 使用案例

案例 1：Code Review Skill — 团队代码审查规范

统一团队代码风格和审查标准，让 AI 按规范自动审查。

SKILL.md 核心内容：
你是一位严格的代码审查员。审查时请检查： 1. 命名：变量/函数用 camelCase，组件用 PascalCase 2. 错误处理：异步操作必须有 try-catch 3. 安全：禁止硬编码密钥，禁止 eval 4. 输出：按「问题 / 建议 / 严重程度」结构化输出
使用：@.skills/code-review "请审查 src/utils/ 下的所有文件"
案例 2：Dark Mode UI Skill — 深色主题设计规范

让 AI 按设计规范生成一致的深色主题界面。

SKILL.md 核心内容：
生成深色主题 UI 时遵循： - 背景：#0f172a / #1e293b - 主色：#38bdf8（accent） - 强调色：#f43f5e（highlight） - 字体：Inter + JetBrains Mono - 圆角：border-radius 0.5rem - 组件：glass-card 风格（毛玻璃、细边框）
使用：@.skills/dark-mode-ui "做一个设置页面的深色主题"

9 实战 Level 1：习惯打卡器

任务：开发一个每日习惯打卡应用

涉及日期处理、localStorage、统计计算、进度条 UI。

约 40 分钟
核心需求
  • 预设 3–5 个习惯（如早起、运动、阅读），可自定义
  • 每日点击打卡，已打卡显示勾选状态
  • 本周完成率统计，进度条展示
  • 数据存 localStorage，刷新不丢失
  • 深色主题，界面简洁
指引 1：先定义数据结构
"我要做一个习惯打卡器。请先设计：如何存储习惯列表？如何存储每日打卡记录（日期+习惯ID）？ 给出数据结构（如 JSON 格式）和文件结构，不要写代码。"
指引 2：Agent 分步实现
"按上面的方案实现。先做 HTML+CSS 骨架和习惯列表，再做打卡逻辑和 localStorage，最后加本周统计和进度条。"
💡 卡住时： 直接贴「当前代码 + 报错信息」给 AI，或问「如何用 JavaScript 获取本周的日期范围？」

10 实战 Level 2：AI 产品落地页 ⭐ 核心实战

任务：为假想 AI 产品做一个完整落地页

比作品集更真实：Hero、功能展示、定价、FAQ、CTA，可部署上线。

约 60 分钟
核心模块
Hero：产品名 + 一句话价值主张 + CTA
功能展示：3–4 个功能卡片，图标+标题+描述
定价：免费/Pro 对比，或单一定价
FAQ：手风琴或折叠，3–5 个问题
Footer：联系、版权、社交链接
阶段 1：写 Spec，让 Agent 规划 (5 min)
"我要做一个 AI 写作助手的落地页。包含：Hero、4 个功能卡片、定价表、FAQ、Footer。 请先给出页面结构（section 划分）和文件组织，不要写代码。"
阶段 2：Sub-Agent 分步执行 (35 min)
  • 🤖 Sub-Agent A: HTML 骨架 + 各 section 占位
  • 🎨 Sub-Agent B: Tailwind/CSS 样式，响应式断点
  • ⚡ Sub-Agent C: FAQ 折叠、导航锚点、平滑滚动

每步完成后审查 diff，必要时用 Cmd+K 微调。

阶段 3：部署 (10 min)
"生成 Vercel 部署配置，并告诉我如何一键部署。"
💡 教练点评： 你的角色是 产品经理 + 架构师 + 验收官。AI 写代码，你负责「要什么」和「对不对」。

11 高级技巧与疑难杂症

🐛 当 AI 写出 Bug？

场景：按钮点击没反应、页面白屏、数据不保存等。

"点击提交按钮没有反应，帮我 debug。这是浏览器控制台的报错：[粘贴报错]"

→ 直接贴报错 + 相关代码片段，Agent 会自动定位并修复。不要只说「有问题」。

📚 上下文窗口溢出？

症状：AI 开始「忘记」之前说的、重复生成旧代码、答非所问。

  • 开新 Chat/Agent 会话，不要硬撑
  • 把关键信息（Spec、架构图）重新粘贴
  • 用 @file 引用具体文件，别让 AI 猜

🎯 AI 理解错意图？

策略：更具体、给参考、给反面。

  • 更具体：「不是整个页面居中，是标题文字居中」
  • 给参考：拖入一张你想要的设计截图
  • 反面例子：「不要用弹窗，用页面内嵌的方式」
